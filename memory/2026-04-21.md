## Architecture Notes

### User Commitment Flow
1. Member receives invite: `t.me/project_bot?start=claim_{projectId}`
2. Telegram opens chat, sends `/start claim_xxx`
3. Bot shows "Yes, I'm in" commitment screen
4. Member taps button ‚Üí `onboardingStatus = 'committed'` in DB
5. Bot can now send DMs to that user

**Telegram rule:** Bot cannot DM a user who has never messaged it first. The `/start claim_xxx` click counts.

### Admin Flow
1. `/create` on @Zolara_builder_bot ‚Üí steps ‚Üí project created ‚Üí bot created
2. Admin shares invite link with team
3. Admin adds project bot to group ‚Üí `my_chat_member` update ‚Üí DM to admin asking to confirm group
4. `/startround [topic]` ‚Üí triggers round ‚Üí members get DM questions ‚Üí synthesis ‚Üí report posted to group

### Group Auto-Detection
When project bot is added to a group: `my_chat_member` update fires ‚Üí looks up project via `botTelegramId` ‚Üí gets admin Telegram ID ‚Üí sends DM asking to confirm group ‚Üí on confirm, `groupIds[]` saved to DB.

### Critical Gap ‚Äî Group Setup in Initiation Flow
`group_setup` step currently asks admin to paste a group ID. This is fragile. The `my_chat_member` approach above is the correct one ‚Äî admin adds bot to group, bot auto-detects and asks for confirmation. The `group_setup` step should be removed/deprecated in favor of this.
